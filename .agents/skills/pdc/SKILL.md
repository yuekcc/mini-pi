# PDCA 轻量版

Plan-Do-Check-Action (PDCA 循环) 以 **验收标准前置** 为原则，将需求从提出到复盘的全过程，固化为可追踪、可验证的文档流和任务流，强调“先定义‘怎样算做完’，再动手实现”。

整个流程是一次完整的 **P-D-C-A 循环**，并在异常时回滚。

此为 PDCA 流程轻量版。

## 术语

- **PRD** → Product Requirements Document，产品需求文档。可以是完整文档，也可以是“一句话需求”，是流程的**唯一输入**
- **task-list.md** → 可执行的任务清单，列出所有任务的名称、简要说明与顺序
- **system-check.md** → 系统级验收清单，从用户/产品视角定义“全部做完且正确”的验收条件
- **二次确认** → 任务执行完成后的检查动作，分两种：**开发复核**（DO 阶段）和**验收确认**（CHECK 阶段）。轻量级流程，二次确认全部由 AI 自主完成。

## 1. PLAN

**目标**：将 PRD 转化为可执行、可验收的任务体系。

**输入**：PRD（产品需求文档）。

**输出物**：

- `task-list.md`：
   - 有序任务列表（任务 ID + 名称 + 依赖关系 + 自测方法 + 状态），可只包含一个任务。状态流转：待开始 → 进行中 → 待验收 → 验收通过 / 验收失败
   - 鼓励在在 task-list 上方增加一段 **方案思路**。
- `system-check.md` → 系统验收条件清单（从用户视角定义功能表现、边界情况等；必要时可包含性能指标）。

**动作**：

1. 分析需求，完成系统设计，将设计拆解为独立可测的任务，写入 task-list.md。
2. 提取出系统级验收条件，写入 system-check.md。
3. 自评 task-list.md 与 system-check.md 的覆盖度和可执行性，确认无遗漏、标准可度量后进入 DO。

## 2. DO

**目标**：按 task-list.md 逐项完成实现与自检。

**动作**：

1. 按 task-list.md 顺序执行任务。每个任务执行时：
   - 对照 task 内容进行编码/配置/文档。
   - **执行 task 内定义的验证方式**（跑测试、人工检查、Code Review 等），确保自检通过。
   - 在 task-list 上标记状态（如：待开始 → 自检完成）。
2. 全部任务自检完成后，执行 **开发二次确认**：重新读取 task-list.md 进行核对，确认无遗漏、无“假完成”。
3. 确认通过后，将代码/产物提交至待验收状态，进入 CHECK 阶段。

**关键原则**：

- **质量内建**：不要把所有测试留到 CHECK 阶段。task 里的验证就是小循环里的 “Check”。

## 3. CHECK

**目标**：以用户视角验收全部功能，确保符合预期。

**动作**：

1. 执行 `system-check.md` 上的所有验收条目。
2. 核对每个 task 的完成状态（验收确认），确保与 `task-list.md` 标记一致。
3. 验收结果：
   - **全部通过** → 进入 Action 阶段。
   - **部分失败或存疑** → 记录失败项，形成问题清单，**回退至 Do 阶段**修复，严重设计偏差则回退至 Plan 阶段调整设计。问题清单直接追加至 task-list.md 尾部，并在回退时相应更新 task-list.md（增加修复任务、标记失败任务）和 system-check.md（补充边界用例），确保文档始终反映最新状态。
4. 回退修复后，**重新走 Check** 流程，直至全部通过。

## 4. ACTION

**目标**：总结经验，闭环 PDCA。

**动作**：

1. 确保问题清单中的条目均已关闭或明确转为后续任务，避免遗留。
2. 总结输出一句话经验，特别是功能实现或修 Bug 过程中发现了规避模式或重要经验。
